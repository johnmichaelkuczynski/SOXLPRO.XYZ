What's different about this app — SOXL Analysis Platform's defining features
This is the second SOXL-focused quant app in your suite (the first was soxl-options-edge). They share a domain but differ in architecture and what's testable. Six things make this one distinct from previous R1 targets:
It has a "never refuse" architectural commitment. Per blueprint, every simulator/engine returns valid (possibly trivial) results for empty/1-bar/non-overlapping windows. R1 must explicitly verify this — submit pathological inputs and confirm the app doesn't crash but produces sensible degraded output (zero-filled rows, captioned as such).
It has a custom Streamlit JavaScript component (components/chart_draw/index.html) for drag-to-draw trend lines. R1 can't easily drag lines via Playwright but can verify the component loads and renders.
Asymmetric resize is a core modeling commitment. The call-sleeve simulator sells DOWN freely but only refills at roll events. R1 verifies this empirically by inspecting the sleeve-value time series.
Multiple LLM-driven features (AI Strategy Builder + Natural-Language Custom Strategy refiner). Both use Anthropic via Replit Integrations. Both produce structured JSON that downstream code must consume. R1 tests both as contracts.
Custom Strategy has a SIGNAL_VERSION constant for cache invalidation. R1 verifies saved strategies survive a session but the system would gracefully handle version mismatches (this is documented behavior — R1 doesn't have to mutate the version, just verify the version is exposed).
It uses both EODHD and Polygon. Polygon for options, EODHD for equity history, plus yfinance for benchmarks. R1 doesn't test the data providers directly but verifies that data loads — and gracefully handles when one is unavailable.
The eight invariants that elevate to CRITICAL VIOLATION status:
Invariant A — Allocation stays within floor/ceiling bounds. Per blueprint, soxl_allocation_engine clips allocation = clip(raw_allocation × regime_mult, floor, ceiling). R1 runs the Allocation Engine, inspects the deviation/allocation panel, and verifies every row's allocation value is in [floor, ceiling]. Hard floor 0.02 and ceiling 0.98 are documented. Any row outside that hard band even with custom floor/ceiling settings = CRITICAL VIOLATION (clipping broken).
Invariant B — No look-ahead in call-sleeve sim. Per blueprint: target_alloc = alloc_df["allocation"].iloc[max(i - 1, 0)]. R1 verifies this by checking that the allocation panel and the realized strategy returns at any timestamp T use the signal computed from data ≤ T-1. R1 can't directly inspect the code path, but R1 can verify: when R1 runs a sim ending at date D, the equity at D should reflect a signal computed before D. R1 detects look-ahead violations by running the sim, then re-running with end_date = D-1, and verifying the equity at D-1 is the same in both runs (if it differs, the original run was peeking into data after D-1). If detected, CRITICAL VIOLATION.
Invariant C — Asymmetric resize is observable in sleeve value series. Per blueprint: sleeve sells down freely, refills only at roll events. R1 inspects the sleeve_value time series + roll_events list + sleeve_alloc_target series. R1 verifies:

Between consecutive roll events, when sleeve_alloc_target increases, sleeve_value does NOT increase proportionally (no mid-cycle refill).
When sleeve_alloc_target decreases (e.g., signal shifts), sleeve_value decreases proportionally (sell-down works).
At roll events, sleeve_value resets to match the new target.
If R1 observes a non-roll-event sleeve_value increase tracking a target increase, the asymmetric rule is broken: CRITICAL VIOLATION.

Invariant D — Probability Engine outputs include sample size. Per blueprint: "Reports n (sample size) alongside every probability." R1 runs the Probability Engine, captures the output. Every reported P(move ≥ M% within H days) row must include an n field. If absent, CRITICAL VIOLATION (the documented spec contract is broken; probabilities without sample sizes are unfalsifiable).
Invariant E — Risk Metrics panel always renders. Per blueprint: "REQUIRED PANEL — always rendered by UI" + compute_risk_metrics returns valid zero-filled row when len(equity) < 2. R1 tests this by deliberately running a backtest with start == end (zero-day window). The Risk Metrics dataframe must still render with all required columns (Total Return %, CAGR %, Vol (ann) %, Max Drawdown %, Sharpe, Sortino, Calmar, Capital at Risk %, Return / At-Risk %), even if values are zeros. Missing dataframe or thrown exception = CRITICAL VIOLATION.
Invariant F — Dislocation Z-score classification matches thresholds. Per blueprint _classify(z): 5 buckets (RICH, STRETCH-RICH, FAIR, STRETCH-CHEAP, CHEAP). The exact thresholds aren't in the blueprint, but the classification function is deterministic. R1 runs the dislocation tab, captures Z value + verdict. Re-runs with different windows. Verifies that the verdict bucketing is monotonic with Z (more positive Z → more RICH; more negative Z → more CHEAP). If the relationship is non-monotonic or randomly assigned, CRITICAL VIOLATION.
Invariant G — AI Strategy Builder produces parseable JSON. R1 sends a substantive query to the AI Strategy Builder. Captures the response. Runs parse_strategy_json mentally: the JSON must include the documented schema (tranched buy ladder, operating rules, statistical basis, risk notes). If the response is unparseable, missing required sections, or contains invented indicators not in ALL_INDICATORS, CRITICAL VIOLATION.
Invariant H — Natural-Language Custom Strategy produces valid spec. R1 enters plain-English in the Custom Strategy NL refiner. The returned JSON must normalize via strategy_nl.normalize_strategy() into something simulate_custom_strategy can consume. R1 verifies by:

Receiving the JSON
Checking for required top-level fields (entry_panel, exit_panel, controls)
Checking that every indicator name in the rules is in the documented ALL_INDICATORS catalog
Checking operators are in OPERATORS = [">", "<", "=", "crosses above", "crosses below"]
If the LLM hallucinates an indicator name not in the catalog (which the blueprint warns about), CRITICAL VIOLATION.
